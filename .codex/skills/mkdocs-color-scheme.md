# Skill: カラースキーム（配色）ベストプラクティス（カード型カタログ向け）

目的：カード型の一覧→詳細を中心とする静的サイトで、読みやすく一貫性があり、拡張しても破綻しにくい配色を**設計/レビュー**する。

このスキルが想定する成果物（例）
- 既存配色の問題点の指摘（可読性、階層、状態色、ダークモード）
- 具体的な「色トークン（CSS変数）」設計案
- ページ/コンポーネント別の適用方針（カード、タグ、CTA、リンク、罫線）
- リリース前チェックリスト

## 使うとき（トリガー）
- 「サイトの配色を整えたい」「カラースキームを見直したい」「ダークモードが眩しい/読みにくい」
- 「カード一覧を見やすくしたい」「タグの色が増殖して困っている」
- 「Material for MkDocsのpaletteやCSS上書き方針を決めたい」
- 「アクセシビリティ（コントラスト等）を満たしたい」

## 使わないとき（非スコープ）
- ロゴ/イラストの制作や画像編集（別途デザイン制作ワークフロー）
- ブランド刷新のようなゼロからのVI設計（リサーチ/ブランド戦略が必要）
- すでにデザインシステムがあり配色が厳密に決まっている場合（その仕様に従う）

## 事前にユーザーから得たい入力（不足していても進める）
- いまの配色（スクショ、CSS、mkdocs.ymlのpalette設定など）
- 目指す雰囲気（落ち着き/ポップ/コントラスト強め等）
- 使いたい基準色（会社カラー等）があればHEX
- 想定ユーザー（スマホ中心か、暗所利用が多いか等）

不足している場合は、まず「保守的なニュートラル＋アクセント1色」案を提示する。

## 手順（設計/レビューの定石）

### 1) 目標を固定する（配色で達成したいこと）
- 可読性：本文・UI・カード情報が一目で読める（コントラスト不足を避ける）
- 階層表現：タイトル/メタ/タグ/ボタンの優先度が自然に伝わる
- 一貫性：同じ意味は同じ色（リンク、警告など）
- アクセシビリティ：色覚差・暗所/明所・モバイルでも破綻しない
- 運用性：色を増やさない（保守コストを抑える）

### 2) 色数を制限する（最重要）
基本は **3系統＋状態色**で足りる。
- Base（背景・面）：背景/カード面/区切り
- Text（文字）：本文/見出し/補助
- Accent（強調）：リンク/CTA/選択状態
- Status（状態）：success / warning / danger（必要最小限）

「見た目が足りないから色を増やす」のは原則NG。まず余白・太字・サイズ・アイコンで整理する。

### 3) まず明度（L）で階層を作る
カード型一覧は色相より**明度差**が効く。
- 背景（最も暗/明）
- カード面（背景より少し浮かせる）
- 罫線/境界（さらに弱く）
- 本文/補助文（濃淡で階層）

最初はニュートラル（グレー）を軸にし、アクセントは後から。

### 4) アクセシビリティの最低ライン
- 通常テキスト：コントラスト比 **4.5:1** 以上
- 大きい文字（見出し等）：**3:1** 以上
- 色だけで状態を伝えない（ラベル文字、アイコン、枠線を併用）
- ダークモードは「反転」ではなく「再設計」
  - ダーク側はアクセントを少し暗く/彩度を落としてネオン化を避ける

### 5) カード/タグ/CTAの配色ルールを固定する

#### 5-1. カード情報密度が高い場合
- タイトル：最も濃い
- メタ情報（人数/時間/年/作者等）：1段薄い
- 外部リンク（例：BGG）：アクセント
- 罫線：主張させない（弱いニュートラル）

#### 5-2. タグ（カテゴリ）色
- 固定パレット（最大8〜10色）で**増やさない**
- 可能なら「色相の役割」を決める（例：難度=青系、時間=橙系…）
- タグは「淡い背景＋濃い文字」で可読性優先（背景を濃くしない）

#### 5-3. 主要CTA（例：『サマリーを見る』）
一覧で繰り返されるボタンは、常時強いと疲れる。
- 通常時：面と馴染む
- hover / focus-visible：明確に強調
- 形・余白・影も併用（色だけに頼らない）

### 6) 実装：色はトークン（変数）で管理する
生のHEXを散らさず、**意味ベースのトークン**で統一する。

推奨トークン（最小構成）
- `--bg`, `--surface`, `--border`
- `--text`, `--text-muted`
- `--accent`, `--accent-hover`, `--accent-contrast`
- `--success`, `--warning`, `--danger`

実装方針（安全順）
1. テーマ設定で可能な範囲（Materialの`palette`など）
2. 追加CSS（変数中心）
3. テンプレ上書き（最終手段）

### 7) フォーカスリングを必ず確認する
キーボード操作の視認性は壊れやすい。
- `:focus-visible` の輪郭/影が、背景と十分に区別できること

## リリース前チェックリスト（5分）
- [ ] 本文が十分に読める（薄いグレー文字になっていない）
- [ ] リンクと通常テキストが区別できる（色だけでなく下線/hoverも）
- [ ] タグの文字が潰れない（淡色背景＋濃い文字）
- [ ] ボタンが「押せる」と分かる（形・余白・hover）
- [ ] ダークモードでアクセントが派手すぎない/眩しくない
- [ ] キーボードフォーカスが見える
- [ ] 重要情報（人数/時間/年/作者）が階層的に読める

---

## よくある失敗 → 回避策
- 色を増やして賑やかにしようとする → 余白・ウェイト・サイズ・区切りで整理してから色
- 全カードのボタンが強すぎて疲れる → 通常は控えめ、hover/focusで強調
- ダークでリンクがネオン化 → ダーク用にアクセントの明度/彩度を下げる
- タグ色が増殖して管理不能 → 最大色数を決め、分類は文字で、色は補助
